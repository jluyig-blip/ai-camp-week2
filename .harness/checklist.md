# 서비스 기획 체크리스트 — 홈페이지/웹서비스 만들 때 고민해야 할 8차원

> AI CAMP 1주차 교안의 "도구 켜기 전에 진단 → 3요소로 구조화 → 타겟 1개 선정" 프레임을
> 서비스 기획에 그대로 적용한 체크리스트. 새 서비스를 시작할 때 이 8개 차원을 차례로
> 짚어가며 답을 채우면 spec.md 가 된다.

에이전트(Claude Code / Codex)에게:
> "checklist.md 의 8차원을 차례로 짚으면서 내 서비스에 맞게 답을 채워줘.
>  한 차원 끝날 때마다 나한테 확인받고 다음 넘어가. 결과는 spec.md 에 저장."

---

## A. 목적·전략 (Why)
1. **타겟 유저** — 누구를 위한 서비스인가?
2. **가치 제안** — 이 서비스가 푸는 통증·만드는 가치는?
3. **성공 지표** — 무엇을 보면 잘되고 있다고 말할 수 있나?
4. **MVP 범위** — *안 할 것*을 명시 (해야 할 것보다 안 할 것이 더 중요).

## B. 정보구조·콘텐츠 (What)
1. **사이트맵** — 어떤 페이지가 어떻게 연결되나?
2. **콘텐츠 모델** — 어떤 데이터 타입이 있나? (게시물·과제·자료·댓글…)
3. **작성 주체** — 누가 콘텐츠를 만드나? (운영자·사용자·AI 에이전트)
4. **거버넌스** — 초안→검수→발행→수정/삭제의 상태머신은? 누가 권한을 가지나?

## C. 사용자·계정 (Who)
1. **인증 방식** — 이메일/비번? 소셜? 매직링크? 토큰?
2. **역할·권한** — 게스트·일반·운영자·관리자. 누가 뭘 할 수 있나?
3. **온보딩 플로우** — 가입부터 첫 가치 경험까지 몇 단계?
4. **데이터 격리** — 멀티테넌시 단위는? (그룹/조직/코호트 등)

## D. 운영·관리자 (Operate)
1. **관리자 페이지** — 운영자가 뭘 보고 뭘 바꾸나?
2. **편집 가능성** — 코드 배포 없이 콘텐츠/설정을 바꿀 수 있나?
3. **로그·모니터링** — 누가 뭘 했는지 추적되나? 에러 알림은?
4. **알림** — 운영자에게 / 사용자에게 어떤 알림이 가나?

## E. 통합·자동화 (Extend)
1. **외부 통합** — 결제·이메일·Slack·분석 등 어떤 외부 시스템과 연결되나?
2. **API · 에이전트 접근** — 봇·CLI 가 사용자 대신 행동할 경로 있나? (이 라이브러리의 `agent-cli-companion` 패턴)
3. **백그라운드 작업** — cron·큐·비동기 처리는?
4. **콘텐츠 파이프라인** — 수집 → 가공 → 발행 자동화 흐름은?

## F. 디자인·UX (Look)
1. **디자인 토큰** — 색·타이포·간격·반경·그림자. (`tokens.json` 기반 + 확장)
2. **컴포넌트 시스템** — 재사용 UI 단위. (`components/catalog.md` 참고)
3. **반응형·접근성** — 모바일·키보드·스크린리더 대응. (`a11y-and-specs.md`)
4. **상태 표현** — 로딩·빈 상태·에러·성공의 일관성. (`state-matrix.md`)
5. **카피·톤** — UX 라이팅 원칙. (`content-voice.md`)

## G. 인프라·보안 (Run)
1. **호스팅·배포** — Vercel/Cloudflare/자체 서버?
2. **DB·스토리지** — Supabase/Postgres/S3?
3. **환경변수·시크릿** — `.env` 관리, CI/CD 변수, 회전 정책.
4. **보안** — RLS·인증 토큰·sanitize·rate limit. **에이전트에게 권한을 줄 때 특히.**
5. **비용·확장성** — 동시 사용자·트래픽 한계·비용 모델.

## H. 라이프사이클 (Evolve)
1. **버저닝·마이그레이션** — DB 스키마 변경 절차.
2. **피처 플래그·점진 롤아웃** — 새 기능을 일부만 켤 수 있나?
3. **사용자 피드백 루프** — 인앱 피드백·설문 채널.
4. **분석·실험** — 누가 뭘 쓰는지 측정, A/B 실험.

---

## 답을 채우는 순서

1주차 교안의 2x2 매트릭스처럼 **모든 차원을 동시에 채우지 말 것.** 순서:

1. **A 먼저** — 누구를·뭘 위해·언제까지·뭘 안 할지. 여기서 막히면 나머지가 다 의미 없음.
2. **C → B** — 누가 쓰나(C)부터 → 그들이 어떤 콘텐츠를 다루나(B).
3. **F·G** — A/B/C 답이 차면 디자인·인프라는 자연스럽게 좁아짐.
4. **D·E** — 운영·통합은 v0 이후 점진 확장.
5. **H** — MVP 시점엔 마이그레이션 절차만 잡고 나머지는 v1 이후.

## 8차원 답을 다 채우면

`.harness/instances/{프로젝트}/README.md` 에 "채택한 공통 패턴" 체크박스 + 확장 토큰 기록.
이게 본인 프로젝트가 design-library 의 한 instance 로 등록되는 결과물.
