# 자동화 프로덕트 UX 원칙 8개

> AI·LLM 기반 자동화 프로덕트를 만들 때 반복적으로 옳았던 설계 원칙.
> 각 원칙은 "왜"와 "구체적 반영 방식"을 같이 적는다.

---

## 1. 사람 개입 지점을 숨기지 말고 드러내라

**왜**: AI가 100% 자동으로 끝내는 일은 거의 없다. 개입 지점이 숨으면 사용자가 "지금 AI가 멋대로 하고 있는지, 내가 승인해야 하는지" 혼란에 빠진다.

**반영**:
- 주요 단계마다 승인 게이트 (예: Director 기획서 승인 → 컴포넌트 생성 시작)
- 현재 단계를 상단 스테퍼로 상시 노출
- "승인 대기" / "생성 중" / "사람 확인 필요" 같은 상태를 색상 배지로

---

## 2. 생성 과정을 시각화하라 (블랙박스 금지)

**왜**: LLM 호출이 여러 단계 병렬로 도는데 진행 중에 아무 것도 안 보이면 대기가 불안해지고, 에러 발생 시 어디서 터졌는지 모른다.

**반영**:
- WebSocket 실시간 이벤트 스트림
- Kanban 뷰로 "생성중 / 검증중 / 통과 / 플래그" 상태 분류
- 각 컴포넌트 카드에 생성자·검증자 모델 명시
- 실패 시 빨간 에러 배너 즉시 노출 (로그에 묻히지 않게)

---

## 3. 플래그·경고는 "무엇/어디/어떻게" 3요소 필수

**왜**: AI가 "이건 이상합니다"만 말하고 끝나면 사람이 10분 더 일해야 한다. "어느 섹션의 어떤 필드가 왜 잘못됐고 어떻게 확인하라"까지 있어야 즉시 행동 가능.

**반영**:
- 모든 플래그 카드에 reason(무엇), location_path(어디), guide(어떻게) 3필드
- 플래그 클릭 → 해당 영역으로 스크롤 + 시각 앵커 (⚐ 마커 + 하이라이트)
- 기술 ID는 접을 수 있게 (raw rubric_id, file path 등)

---

## 4. 재생성은 "가능한 가장 작은 단위"부터

**왜**: 뭔가 잘못됐을 때 전체를 다시 돌리기는 비용·시간이 크다. 작은 단위 재생성이 빠르고 정확하다.

**반영**:
- 재생성 scope 3종: **item(문항 1개)** / **chapter(학습자료+퀴즈+실습 한 세트)** / **component(퀴즈 전체)**
- 기본은 item 단위. 대규모는 명시적 선택.
- 재생성 시 "시작 지점"과 "끝 지점"을 명확히 (검증 통과 / 3회 재시도 실패 / 사용자 수락)
- 재생성 시 원본 컨텍스트(기획서·형제 항목·피드백) 자동 상속

---

## 5. 색상을 시맨틱으로 잠그라 (일관성)

**왜**: 같은 빨강이 어떨 땐 "삭제", 어떨 땐 "중요", 어떨 땐 "플래그 중 하나"면 학습이 안 된다.

**반영**:
- Status 색상 ≠ Flag 색상 ≠ Action 색상
- Status: 생성중=blue / 검증중=amber / 통과=emerald / 플래그=rose / 재생성=amber
- Flag 5종은 각자 고유 색 (FACT=rose, SCHEMA=amber 등)
- Toast: 성공=emerald / 오류=rose / 정보=zinc
- Anchor 하이라이트: amber 고정 (플래그 타입과 무관)

---

## 6. 입력을 Tier로 나눠라 (필수 / 품질 / 운영)

**왜**: 사용자가 처음 보는 화면에 20개 필드가 있으면 포기한다. 근데 필요할 때는 다 접근 가능해야 한다.

**반영**:
- **Tier 1 (필수)**: 매 run마다 입력하는 것 — 위인·주제·학습대상 등 5개 이하
- **Tier 2 (품질)**: 슬라이더·드롭다운 — 기본값 제공, 필요 시 조정
- **Tier 3 (운영)**: 접어둔 고급 설정 — 재시도 상한, 타깃 AI, 필수/금지 키워드
- **프리셋 (관리자만)**: 서식·루브릭·톤앤매너 — 읽기전용 표시

---

## 7. 기술 용어 → 한글 직관 라벨

**왜**: 기획자/검토자는 개발자가 아니다. `material > material.voice.mentor_persona` 같은 raw ID가 노출되면 이해하려 멈춘다.

**반영**:
- 루브릭 ID → 친근한 "영역 · 항목" 명으로 매핑 사전 관리
- 플래그 카테고리: `BORDERLINE` → `애매함 — 사람 확인`
- 컴포넌트 타입: `material` → `학습자료`, `figure_rationale` → `위인 선정 배경`
- raw ID는 접힘(details)에만 표시 — 필요할 때만 열기

---

## 8. "열 수 있게, 조절 가능하게, 지울 수 있게" 편집성 원칙

**왜**: 자동화 프로덕트일수록 사람이 "중간에 끼어들어 조정할 여지"가 필요하다.

**반영**:
- 기획서는 JSON 덤프가 아니라 **편집 가능한 폼** (필드별 input/textarea)
- 레이아웃 패널 폭은 **드래그 리사이저** (3-pane 사이 divider, 테이블 컬럼 핸들)
- 히스토리(run) 개별 **삭제** 버튼
- 결과물은 **여러 포맷 다운로드** (xlsx 전체 / 컴포넌트 개별 docx)
- 모든 편집·삭제는 localStorage로 **사용자 세팅 기억**

---

## 추가 고려: Anti-패턴 (하지 말 것)

- ❌ 생성 완료 후 JSON 덤프 보여주기 (사용자 입장에선 비인간적)
- ❌ "처리 중..." 스피너만 돌리고 진행 세부 안 보여주기
- ❌ 실패 시 콘솔 로그에만 찍고 UI는 무반응
- ❌ 기술 ID를 그대로 UI 라벨로 쓰기
- ❌ 재생성 할 때마다 전체 파이프라인 다시 돌리기
- ❌ 플래그를 "목록"으로만 보여주고 해당 콘텐츠와 연결 안 하기
- ❌ 영문 필터명·용어 그대로 노출
