Playbook · deploy-rollback

severity: high deploy deploy-gate-agent → orchestrator SLA: roll back within 10 min of the call
Summary — A deploy is live and causing user impact (or is about to). Goal: get back to a known-good state fast, then diagnose calmly. Rolling back is cheap; debugging in production is not.

Trigger

  • monitor-agent or a person reports user impact correlated with a deploy that just shipped.
  • The deploy succeeded the gate but a smoke check / golden path is now failing in production.
  • A hotfix is being attempted but isn't ready and the bleeding is ongoing.

Decide: roll back, or roll forward?

is there active user impact? ├─ yes, and a fix isn't <10 min away → ROLL BACK now, diagnose after ├─ yes, but a 1-line revert fixes it → roll forward with the revert (counts as incident-response) └─ no, just elevated errors / noisy logs → no rollback; open a normal task, watch the trend

How to roll back (fill in for your platform)

  1. Identify the last known-good release/deployment (check data/runs/last-gate.json, your hosting dashboard, or git log on the deploy branch).
  2. Promote / redeploy that release.
    • Vercel: redeploy the previous deployment, or vercel rollback <url>.
    • Containers/k8s: re-deploy the previous image tag.
    • Plain git deploy: git revert <bad-commit> (don't force-push the branch) and let the pipeline ship it.
  3. If the bad deploy ran a forward database migration, roll the schema back too (or apply the down migration) — coordinate with backend-agent. Never leave the schema ahead of the code.
  4. Confirm recovery: re-run the golden-path smoke check, watch the error rate fall.
  5. Post the recovery notice in the incident channel.

After the rollback

  • Within 1 hour: register the incident-response task (metadata.source = "incident-response") describing what shipped, what broke, and the rollback.
  • Open a normal task to fix the underlying bug — it must re-pass the gate before re-shipping.
  • If the gate let a breaking change through, file a task to add the check that would have caught it.
  • Write a postmortem if there was user impact.
